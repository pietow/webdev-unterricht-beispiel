- alternative zu useEffect war `TanStack Query` Daten laden über ein API

````tsx
function MyComponent({ personId }) {
    const { data, error, isPending } = useQuery({
        queryKey: ["persons", personId],
        queryFn: () => fetch(`https://..../${personId}`).then((res) => res.json())
    })
    if (loading) {return <div>Loading...</div>}
    if (error) {return <div>Error</div>}
}
````

### Typ-Sicherheit für die API-Response hinzufügen

- Problem: Bei `fetch(...).json()` kennt TypeScript die Struktur der Response nicht zuverlässig, daher wird `data` oft als `any` inferiert.
- Ziel: Die Response so typisieren, dass:
  - TypeScript korrekte Typen erkennt (z.B. für `map(...)`)
  - die Response-Struktur zusätzlich zur Compile-Time-Typisierung auch **zur Laufzeit** geprüft wird

### Option A: Type Assertion

- Man könnte eine TypeScript-Type-Assertion nutzen:

```ts
return response.json() as Promise<{id: number, title: string, description: string}[]>
```

- Nachteil:
  - Das ist nur eine Annahme über das Response-Schema.
  - Funktioniert gut, wenn du die API selbst kontrollierst.
  - Riskant, wenn:
    - ein anderes Team die API entwickelt
    - es eine Third-Party-API ist

  - Denn: Type Assertions prüfen nichts zur Laufzeit. Wenn die API etwas anderes liefert, bekommst du potenziell Runtime-Fehler trotz "grüner" Typen.

### Option B: Zod Schema Validation (empfohlen)

- Statt einer bloßen Annahme wird die Response mit **Zod** validiert.
- Vorteil:
  - Schema wird **zur Laufzeit** überprüft.
  - Bei Abweichung wird ein Fehler geworfen (und TanStack Query kann damit sauber in den Error-State gehen).

- Hier wird dasselbe Zod-Schema verwendet wie bei der SQL-Validierung, weil sich die Struktur nach dem Query-Result nicht verändert.

```ts
export function PostList( ... ) {
```

### Recap der Typ-Safety-Verbesserung

- Route Handlers sind für Data Fetching oft bevorzugt gegenüber Server Functions, weil sie:
  - parallele API-Requests erlauben
  - HTTP GET für Reads nutzen (statt POST)

- Nachteil von Route Handlers:
  - keine automatische Typsicherheit über die Netzwerkgrenze

- Lösung:
  - Zod validiert die Response-Struktur zur Laufzeit und stellt damit praktische Typ-Sicherheit wieder her

### Nächster Schritt

- TanStack Query für **Mutations** verwenden (z.B. neue Posts erstellen, Cache invalidieren, Liste aktualisieren).

### Daten mutieren mit einer TanStack-Query-Mutation

  - Verstehen, wie TanStack Query Mutations funktionieren.
  - `useMutation` im `NewPost`-Component einsetzen.
  - Das Problem lösen, dass neu erstellte Posts nicht automatisch in der Posts-Liste erscheinen.

### Verständnis von TanStack Query Mutations

- TanStack Query stellt den Hook `useMutation` bereit, der den gesamten Mutation-Prozess verwaltet, inklusive:
  - Statusvariablen (z.B. "läuft gerade", "ist fehlgeschlagen")
  - Mechanismen zur Cache-Aktualisierung nach einer erfolgreichen Mutation (z.B. via Invalidation)

### Beispiel: `useMutation` mit Cache-Invalidation

- Mutation ausführen und danach den relevanten Cache invalidieren, damit frische Daten geladen werden:

```ts
const { mutate, isPending, isError } = useMutation({
  mutationFn: (newProduct) => createProduct(newProduct),
  onSuccess: async () => {
    await queryClient.invalidateQueries({
      queryKey: ['products'],
    })
  },
})

async function handleClick() {
  mutate({
    name: 'New product',
    description: 'New product description',
  })
}

if (isPending) return 'Mutating... '
if (isError) return 'An unexpected error occurred'
```

### Wichtige Optionen von `useMutation`

- `useMutation` erhält ein Options-Objekt. In diesem Beispiel relevant:

- `mutationFn`
  - Die Funktion, die den Server-Aufruf für die Mutation ausführt.
  - Im Beispiel: Aufruf einer Server Function `createProduct(newProduct)`.

- `onSuccess`
  - Callback, der nach erfolgreicher Mutation ausgeführt wird.
  - Im Beispiel: Cache für `["products"]` wird invalidiert.
  - Effekt: TanStack Query weiß, dass die alten Daten nicht mehr gültig sind und lädt bei Bedarf eine frische Version.

### Wichtige Rückgabewerte von `useMutation`

- `useMutation` liefert ein Objekt mit u.a. diesen Members:

- `mutate`
  - Startet die Mutation.
  - Im Beispiel wird `mutate(...)` in einem Click-Handler aufgerufen.

- `isPending`
  - Zeigt an, dass die Mutation gerade läuft.
  - Typischer Einsatz: "Mutating..."-Indikator oder Button deaktivieren.

- `isError`
  - Zeigt an, dass die Mutation fehlgeschlagen ist.
  - Typischer Einsatz: Error Alert anzeigen.

### Nächster Schritt

- `useMutation` im `NewPost`-Component implementieren und nach erfolgreichem Create die Posts-Query (z.B. `["posts"]`) invalidieren, damit neue Posts sofort in der Liste erscheinen.

### `useMutation` verwenden

- Ziel:
  - State Management in `NewPost` vereinfachen, indem `useMutation` den Mutation-Status übernimmt.
  - Nach erfolgreicher Erstellung den Posts-Cache invalidieren, damit neue Posts sofort in der Liste erscheinen.

### 1) Lokale `useState`-States entfernen

- Datei: `NewPost.tsx`
- Entfernen:
  - bestehende `isMutating`-State-Logik
  - bestehende `status`-State-Logik
  - Import von `useState` (falls danach ungenutzt)

### 2) `useMutation` importieren und initialisieren

- `useMutation` importieren und im Component aufrufen:

```ts
import { useMutation } from "@tanstack/react-query";
...
export function NewPost() {
  const { mutate, isPending, isError, isSuccess } = useMutation({
    mutationFn: ({
      title,
      description,
    }: {
      title: string;
      description: string;
    }) => createPost(title, description),
  });

  async function handleClick() {
    ...
  }

  return ...
}
```

- Bedeutung:
  - `mutationFn` ruft die Server Function `createPost(title, description)` auf.
  - `mutate` startet die Mutation.
  - `isPending`, `isError`, `isSuccess` sind Statusvariablen, die TanStack Query für dich verwaltet.

- Hinweis aus dem Text:
  - Man könnte Mutations auch über Route Handler lösen.
  - Server Functions sind hier aber einfacher und bieten mehr Typsicherheit.

### 3) `mutate(...)` im Click-Handler verwenden

- Der Handler braucht kein `async` mehr.
- Alter State-Code wird entfernt.
- Neuer Handler:

```ts
function handleClick() {
  mutate({
    title: 'New Post',
    description: 'New Post Description',
  })
}
```

### 4) JSX auf `isPending`, `isError`, `isSuccess` umstellen

- Ersetze:
  - `isMutating` -> `isPending`
  - Status-Abfragen -> `isError` und `isSuccess`

```tsx
<div ... >
  <button ... >
    {isPending ? "Creating..." : "Create New Post"}
  </button>

  {isError && (
    <span ... >An unexpected error occurred</span>
  )}

  {isSuccess && (
    <span ... >Post successfully created</span>
  )}
</div>
```

- Verhalten an dieser Stelle:
  - Button/Status funktionieren wie zuvor.
  - Aber: Neuer Post erscheint noch nicht automatisch in der Liste, weil die Posts-Query im Cache nicht invalidiert wurde.

### 5) Posts-Cache nach Erfolg invalidieren (`useQueryClient`)

- `useQueryClient` importieren und `invalidateQueries` in `onSuccess` verwenden:

```ts
import { ..., useQueryClient } from "@tanstack/react-query";
...
export function NewPost() {
  const queryClient = useQueryClient();

  const { ... } = useMutation({
    mutationFn: ...,
    onSuccess: async () => {
      queryClient.invalidateQueries({
        queryKey: ["posts"],
      });
    },
  });

  ...
}
```

- Bedeutung:
  - `useQueryClient()` gibt dir Zugriff auf den globalen `queryClient`.
  - `invalidateQueries({ queryKey: ["posts"] })` markiert die Posts-Daten als "stale".
  - TanStack Query refetcht dann die Posts-Query, wodurch die Liste aktualisiert wird.

- Ergebnis:
  - Nach Klick auf "Create New Post" erscheint der neue Post jetzt in der Posts-Liste.

### Kurz-Recap

- `useMutation` vereinfacht Mutations, weil:
  - Status (`isPending`, `isError`, `isSuccess`) automatisch gemanagt wird
  - weniger eigener State-Code nötig ist

- Mit `QueryClient.invalidateQueries` im `onSuccess`:
  - werden betroffene Cache-Einträge invalidiert
  - und die UI erhält frische Daten passend zur Mutation

### Zusammenfassung

- Einstieg mit client-seitigem Data Fetching über den React-Hook `useEffect`.
- Erkenntnis, dass robuste Implementierungen mit `useEffect` schnell komplex und fehleranfällig werden.
- Wechsel zu **TanStack Query**, wodurch:
  - der Code deutlich vereinfacht wird
  - automatische Retries bei Fehlern möglich sind
  - Lade- und Fehlerzustände zentral gemanagt werden

### Vorteile von TanStack Query im Client

- Automatisches **Refetching**, wenn der Browser-Tab den Fokus verliert und wieder erhält.
- Dieses Verhalten ist mit rein server-seitigem Data Fetching nicht möglich.
- Deutlich bessere User Experience ohne zusätzlichen Implementierungsaufwand.

### Server Functions vs. Route Handlers

- Server Functions haben Nachteile für client-seitiges Data Fetching:
  - Nutzung von HTTP POST statt HTTP GET
  - sequentielle Ausführung

- Umstellung auf **Next.js Route Handlers**, um:
  - klassische GET-Endpunkte zu nutzen
  - parallele Requests zu ermöglichen
  - Best Practices für Data Fetching einzuhalten

### Mutations mit Server Functions und TanStack Query

- Server Functions eignen sich sehr gut für **Mutations**:
  - einfache Implementierung
  - gute Typsicherheit

- Verwendung einer Server Function zum Erstellen neuer Blog-Posts.
- Kombination mit **TanStack Query Mutations**, um:
  - Mutations-Zustände (`isPending`, `isSuccess`, `isError`) zu verwalten
  - nach erfolgreicher Mutation den Cache zu invalidieren

- Erkenntnis: Nach Mutations muss der TanStack-Query-Cache explizit invalidiert werden, damit Listen aktualisiert werden.





| Use Case                           | **Route Handler**                           | **Server Function**                              |
| ---------------------------------- | ------------------------------------------- | ------------------------------------------------ |
| Data Fetching (Reads)              | Häufig bevorzugt (GET, parallel, klare API) | Möglich, aber oft POST + Sequenzialität          |
| Mutations (Writes)                 | Gut (REST-Style, klare Endpunkte)           | Sehr gut (einfach, typsicher, wenig Boilerplate) |
| Typ-Sicherheit über Netzwerkgrenze | Nicht automatisch, typischerweise mit Zod   | Besser (typ-safe boundary)                       |
