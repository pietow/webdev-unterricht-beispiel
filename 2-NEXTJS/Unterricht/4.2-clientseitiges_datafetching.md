## Herausforderungen bei Data Fetching mit `useEffect`

* Mit `useEffect` musst du typischerweise **mehrere Zustände selbst verwalten**:

  * den eigentlichen Datenzustand (z.B. `name`)
  * einen Ladezustand (z.B. `loading`)
* Beispiel (Erinnerung) mit Daten + Loading-State:

```ts
const [name, setName] = useState<string | undefined>();
const [loading, setLoading] = useState(true);
useEffect(() => {
  getPerson().then((person) => {
    setLoading(false);
    setName(person.name);
  });
}, []);
if (loading) {
  return <div>Loading ...</div>;
}
```

### Zusätzliche Komplexität durch Fehlerbehandlung

* Sobald du Error Handling ergänzt, brauchst du **noch mehr State** (z.B. `error`).
* Der Code wird dadurch deutlich verzweigter und fehleranfälliger:

```ts
...
const [error, setError] = useState<Error | undefined>();
useEffect(() => {
  getPerson()
    .then((person) => {
      ...
    })
    .catch(e => {
      setError(e);
      setLoading(false);
    });
}, []);
...
```

### Noch komplexer mit Parametern und Refetching

* In realen Apps brauchst du meist Parameter wie eine ID (z.B. `personId`).
* Dann muss `useEffect` von dieser ID abhängen:

```ts
useEffect(() => {
  getPerson(personId)
    ...
}, [personId]);
```

* Konsequenzen dieser Abhängigkeit:

  * Bei jeder Änderung von `personId` wird automatisch ein **Refetch** ausgelöst.
  * Es kann zu **Race Conditions** kommen, wenn `personId` sich ändert, während ein Fetch noch läuft.
  * Wenn die Komponente **unmounted**, während ein Fetch noch läuft, kann es zu Problemen kommen, wenn danach versucht wird, State zu setzen (klassisches "setState on unmounted component"-Problem).

### Warum das in der Praxis schnell eskaliert

* Sobald du Edge Cases sauber lösen willst (Race Conditions, Cancellation, Retries, Caching, Synchronisation), steigt der Aufwand stark an.
* Verweis auf eine detailliertere Einordnung durch einen TanStack-Query-Maintainer:

```txt
https://tkdodo.eu/blog/why-you-want-react-query
```

### Verständnis von TanStack Query

* TanStack Query vereinfacht Data-Fetching-Code, indem es typische Zustände für dich verwaltet, z.B.:

  * Loading-Zustände
  * Error-Zustände
* Kernidee: Ein intelligenter **Data Cache** für bereits geladene Daten.
* Wenn gecachte Daten **"stale"** (veraltet) werden, führt TanStack Query automatisch einen **Refetch** aus, um die Daten zu aktualisieren.

### Beispiel: `useQuery` in einer Komponente

* Die Komponente lädt ein Produkt (per `id`) und rendert Name und Beschreibung.
* TanStack Query übernimmt dabei die Zustandslogik für Laden/Fehler/Daten:

```ts
function Product({ id }: { id: number }) {
  const { data, error, isPending } = useQuery({
    queryKey: ["products", id],
    queryFn: () =>
      fetch(`https://some-server.com/products/${id}`).then((res) => res.json()),
  });
  if (isPending) return "Loading... ";
  if (error) return "Error: " + error.message;
  return (
    <div>
      <h2>{data.name}</h2>
      <p>{data.description}</p>
    </div>
  );
}
```

### `useQuery`: Zweck und wichtigste Optionen

* TanStack Query stellt den Hook `useQuery` bereit, der den gesamten Fetch-Prozess steuert.

* Konfiguration erfolgt über ein Options-Objekt; zwei zentrale Optionen sind:

* `queryKey`

  * Ein **eindeutiger Schlüssel** für die Daten im Cache.
  * Da der Cache viele unterschiedliche Query-Ergebnisse speichern kann, identifiziert der Key die Daten eindeutig.
  * Im Beispiel: `["products", id]` (Kategorie + konkrete Produkt-ID)

* Hinweis zur Best Practice

  * Ein **Array** als Key gilt als Best Practice, weil TanStack Query damit einzelne Teile des Keys auswerten kann.
  * Beispielhafte Konsequenz: Wenn `["products"]` invalidiert wird, werden automatisch auch Keys wie `["products", 1]`, `["products", 2]`, ... invalidiert.

* `queryFn`

  * Die Funktion, die das tatsächliche Fetching durchführt.
  * Im Beispiel wird mit `fetch(...)` ein Produkt vom API-Endpunkt geladen.

### Rückgabewerte von `useQuery`

* `useQuery` liefert ein Objekt mit vielen hilfreichen Werten, u.a.:

* `data`

  * Die geladenen Daten.

* `isPending`

  * Ob die Daten noch nicht geladen sind.
  * Typischer Einsatz: Loading-Indikator.

* `isSuccess`

  * True, wenn die Daten erfolgreich geladen wurden.

* `isError`

  * True, wenn beim Laden ein Fehler aufgetreten ist.
  * Typischer Einsatz: Fehleranzeige/Alert.

* `error`

  * Die `Error`-Instanz, falls ein Fehler aufgetreten ist.

### Voraussetzung: `QueryClientProvider` im Component Tree

* `useQuery` funktioniert nur, wenn es innerhalb eines `QueryClientProvider` verwendet wird.
* Das sorgt dafür, dass die App **einen gemeinsamen Cache** nutzt (statt mehrere isolierte Caches).

```ts
function App() {
  const [queryClient] = useState(() => new QueryClient());
  return (
    <QueryClientProvider client={queryClient}>
      ...
    </QueryClientProvider>
  );
}
```

### Warum `QueryClient` in State gehalten wird

* Der `QueryClient` wird in `useState` gehalten, damit **dieselbe Instanz** auch nach Re-Renders wiederverwendet wird.
* Dadurch bleibt der Cache stabil und wird nicht bei jedem Render neu erstellt.

### Cache-Steuerung mit `QueryClient`

* `QueryClient` kann auch direkt genutzt werden, um den Cache zu steuern.
* Häufiger Use Case: **Invalidation**, damit frische Daten neu geladen werden.

```ts
queryClient.invalidateQueries({ queryKey: ["products"] });
```

### Recap der Umstellung auf TanStack Query

* TanStack Query vereinfacht client-seitiges Data Fetching durch:

  * automatische Verwaltung von Loading/Error-States
  * Caching
  * Refetching bei "stale" Daten
  * Retries bei fehlgeschlagenen Queries

* Default-Verhalten:

### Route Handler mit TanStack Query verwenden

* In diesem Abschnitt geht es um **Next.js Route Handlers** und warum sie sich für Data Fetching oft besser eignen als **Server Functions**.
* Danach wird die App so umgebaut, dass TanStack Query über **Route Handlers** statt über Server Functions lädt.

### Vorteile von Route Handlers

* Ein Next.js Route Handler ermöglicht dir, einen **API-Endpunkt** bereitzustellen (also eine klassische HTTP-Route, die Daten liefert).

* Die Server-Function-Variante ist zwar einfach und bietet **Typsicherheit über die Netzwerkgrenze**, hat aber relevante Nachteile:

* **Parallelität**

  * Server Functions laufen **sequenziell**.
  * Das typische Aufrufmuster im Komponentenbaum führt oft zu Sequenzialität, weil Calls entlang der Render-Reihenfolge entstehen und nicht automatisch gebündelt/parallelisiert werden.
  * Beispiel: Eine Komponente lädt Daten via Server Function, und eine Child-Komponente lädt weitere Daten via Server Function.
  * Dann muss der zweite Call warten, bis der erste fertig ist.
  * Route Handlers können im Gegensatz dazu **parallel** bedient werden (mehrere Requests können gleichzeitig laufen).

* **HTTP-Methode**

  * Server Functions laufen als **HTTP POST**.
  * Für Data Fetching ist jedoch **HTTP GET** der Standard (POST wird typischerweise für Mutations verwendet).
  * Route Handlers erlauben dir sauber ``GET`` für Reads und ``POST/PUT/PATCH/DELETE`` für Writes zu nutzen.


  * Die Next.js- bzw. React-Teams empfehlen **Server Functions nicht** als primären Weg für Data Fetching.
  * Route Handlers sind hier der etablierteste und kompatibelste Ansatz.

### Konsequenz für die App

* Weil Route Handlers diese Nachteile vermeiden, werden sie jetzt im Projekt eingesetzt, und das Data Fetching wird entsprechend umgestellt.

### 2) Route Handler für `src/app/api/posts/route.ts` erstellen

* Route Handlers liegen unter `src/app/api`.
* Statt `page.tsx` verwendet man `route.ts`.
* Datei: `src/app/api/posts/route.ts`

```ts
import { type NextRequest } from "next/server";
import { getAllPosts, getFilteredPosts } from "@/data/queries";

export async function GET(request: NextRequest) {
  const criteria = request.nextUrl.searchParams.get("criteria");
  if (typeof criteria === "string") {
    return Response.json(await getFilteredPosts(criteria));
  }
  return Response.json(await getAllPosts());
}
```

* Bedeutung:

  * Behandelt GET-Requests auf `/api/posts`.
  * Liest optional `criteria` aus den Query-Parametern.
  * Wenn `criteria` vorhanden ist, werden gefilterte Posts geliefert, sonst alle Posts.

* Test im Browser:

  * Aufrufen: `http://localhost:3000/api/posts`
  * Ergebnis: JSON aller Posts.

### 3) Route Handler für `src/app/api/posts/[id]/route.ts` erstellen

* Datei: `src/app/api/posts/[id]/route.ts`

```ts
import { getPost } from "@/data/queries";

export async function GET(
  _: Request,
  { params }: { params: Promise<{ id: string }> }
) {
  const id = Number((await params).id);
  if (!Number.isInteger(id)) {
    return Response.json({ message: "Post not found" }, { status: 404 });
  }
  const data = await getPost(id);
  if (!data) {
    return Response.json({ message: "Post not found" }, { status: 404 });
  }
  return Response.json(data);
}
```

* Bedeutung:

  * Behandelt GET-Requests auf `/api/posts/[id]`.
  * Extrahiert `id` aus der Route.
  * Wenn `id` nicht numerisch ist oder kein Datensatz gefunden wird:

    * Antwort mit Status `404`.

* Test im Browser:

  * Aufrufen: `http://localhost:3000/api/posts/1`
  * Ergebnis: JSON für Post 1 (sofern vorhanden).

### 4) `PostList` auf Route Handler umstellen

* Datei: `PostList.tsx`
* Änderungen:

  * Imports von `getAllPosts` und `getFilteredPosts` entfernen.
  * `useQuery` so ändern, dass `fetch(...)` gegen den Route Handler läuft.

```ts
const { ... } = useQuery({
  queryKey: ...,
  queryFn: async () => {
    const path =
      typeof criteria === "string"
        ? `/api/posts/?criteria=${encodeURIComponent(criteria)}`
        : "/api/posts/";
    const response = await fetch(path);
    if (!response.ok) {
      throw new Error("Problem fetching data");
    }
    return await response.json();
  },
});
```

* Logik:

  * Wenn `criteria` gesetzt ist, wird `/api/posts/?criteria=...` verwendet.
  * Sonst `/api/posts/`.
  * Bei HTTP-Fehlerstatus wird ein Error geworfen, damit TanStack Query in den Error-State geht.

* Hinweis zur Typisierung:

  * `resolvedPosts` wird hier oft nur als `any` inferiert, daher kann beim `map(...)` ein Type-Problem auftauchen.
  * Funktional läuft es trotzdem; die Typen werden später sauber gemacht.

### 5) `PostDetail` auf Route Handler umstellen

* Datei: `PostDetail.tsx`
* Änderungen:

  * Import von `getPost` entfernen.
  * `useQuery` so ändern, dass `fetch(...)` gegen `/api/posts/${id}` läuft.

```ts
const { ... } = useQuery({
  queryKey: ...,
  queryFn: async () => {
    const response = await fetch(`/api/posts/${id}`);
    if (!response.ok) {
      throw new Error(
        response.status === 404 ? "Blog post not found" : "Problem fetching data"
      );
    }
    return await response.json();
  },
});
```

* Logik:

  * `fetch` ruft den Route Handler mit der `id` auf.
  * Bei Fehler:

    * Status `404` wird speziell behandelt, um eine passende Fehlermeldung zu liefern.
    * Andere Fehler liefern `"Problem fetching data"`.

* Hinweis zur Typisierung:

  * Auch hier wird die Query-Response ohne explizite Typen oft als `any` inferiert.
  * Funktioniert zur Laufzeit, ist aber nicht ideal; wird im nächsten Abschnitt adressiert.
